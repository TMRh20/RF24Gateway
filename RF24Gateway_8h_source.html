161  * @code

165  * The first example entry would route all traffic to the 10.1.4.x subnet to 10.1.3.33 <br>

168  * Data can be accessed using standard linux Internet address manipulation routines as follows:
169  * @code
170  * printf("**IP\t\tMask\t\tGateway**\n");

172  * printf("%s \t",inet_ntoa(gw.routingStruct[i].ip));//inet_ntoa uses a statically assigned buffer, so the printf calls need to be done separately
173  * printf("%s \t",inet_ntoa(gw.routingStruct[i].mask));
174  * printf("%s\n", inet_ntoa(gw.routingStruct[i].gw));
175  * //std::cout << inet_ntoa(gw.routingStruct[i].ip) << " \t" << inet_ntoa(gw.routingStruct[i].mask) << " \t" << inet_ntoa(gw.routingStruct[i].gw) << std::endl;
176  * }
177  * printf("*****\n");

179  * Users can modify the routing table as desired, but changes made in code will not currently be saved to file

181  * @endcode

308 /**
309  * @mainpage RF24Gateway
310  *
311  * RF24Gateway is a library style replacement for the standalone RF24toTUN application, used with RF24Ethernet.
312  *
313  * In addition to passing TCP/IP payloads to a network interface, RF24Gateway also allows users to send and receive
314  * standard RF24Network payloads.
315  *
316  * This allows users to deploy and manage hybrid networks, consisting of nodes communicating via TCP/IP and RF24Network Messages
317  *
318  * @section Installation Installation
319  *
320  * See http://nRF24.github.io/RF24 for installation instructions using the installer, or clone the RF24 libs and run 'make install' for each one.
321  *
322  * After running the installer, RF24Gateway will be installed at rf24libs/RF24Gateway. <b><a href="examples.html">Examples</a></b> are included for operating the gateway
323  * as well as client scripts & programs to control nodes via http requests.
324  *
325  * Note: RF24Gateway is tested with and defaults to using RF24Mesh. Sensor nodes must also use RF24Mesh or be statically assigned within RF24Mesh.
326  *
327  * See http://nRF24.github.io/RF24Ethernet/ for related documentation for use with RF24Gateway.
328  *
329  * @section HowItWorks How It Works
330  *
331  * RF24Gateway is relatively simple in design, and uses the <a href="https://www.kernel.org/doc/Documentation/networking/tuntap.txt">Universal TUN/TAP driver for Linux</a> <br>
332  * Data incoming from the RF24 communication stack (designated as external data) is passed to the interface without modification. Conversely, incoming TCP/IP or other data
333  * received on the TUN/TAP interface is passed directly into the radio network and designated as external data.
334  *
335  * Linux devices running RF24Gateway can establish network links using the built-in TCP/IP stack, and users can include Arduino/AVR devices or additional RPi devices to automatically
336  * extend the wireless range.<br>
337  * Arduino/AVR devices, etc must utilize a software IP stack such as uIP.
338  *
339  *
340  * @section SimpleConfig Simple Configuration
341  *
342  * In the standard configuration, a single Linux/RF24Gateway node is used to provide connectivity to one or more
343  * Arduino nodes running RF24Ethernet.
344  * In this case the master node could be configured with IP 10.1.3.1 netmask 255.255.255.0 and nodeID 0
345  * The remaining Arduino nodes can used nodeIDs 2-253, and would be assigned IP addresses 10.1.3.2-253 and netmask 255.255.255.0
346  *
347  * If there are problems starting the examples, the following commands can be run to configure the interface:
348  *
349  * @code
350  * ip tuntap add dev tun_nrf24 mode tun user pi multi_queue
351  * ifconfig tun_nrf24 10.1.3.1/24 @endcode
352  *
353  * To enable nat and routing, the following commands can also be run:
354  *
355  * @code
356  * sudo sysctl -w net.ipv4.ip_forward=1
357  * sudo iptables -t nat -A POSTROUTING -j MASQUERADE @endcode
358  *
359  * See http://nRF24.github.io/RF24Ethernet/ConfigAndSetup.html for more info
360  *
361  * @section AdvConfig Advanced Configuration
362  *
363  * In a more advanced configuration, users can use two or more nodes running RF24Gateway together, along with Arduino Nodes
364  * running RF24Ethernet
365  * In this case the master node could be configured differently with IP 10.1.3.1 netmask 255.255.0.0 and nodeID 0
366  * The child RPi2 node could be assigned IP 10.1.3.33 netmask 255.255.0.0 and nodeID 33
367  * Another device, lets say usb0 connected to the child RPi2 node is assigned IP 10.1.5.1 netmask 255.255.255.0
368  *
369  * In this configuration, connectivity between all nodes in the mesh would be established, but a routing table is
370  * needed to include attached and internet devices.
371  *
372  * To fully enable routing beyond the RPi devices, a routing table needs to be added to the Master node:
373  * 1. Rename the file RF24Gateway/examples/routing.txx to routing.txt
374  * 2. Edit the file accordingly
375  * 3. Restart the gateway example to reload routing info
376  *
377  * @section NetCfg Network Configuration Example
378  * Local WiFi/Ethernet Network (Windows,Linux,RPi devices etc) - 10.10.1.0 - 255.255.255.0 ( 10.10.1.x network ) <br>
379  * RPi devices (tun_nrf24 interfaces & all connected devices) - 10.1.0.0 - 255.255.0.0 ( 10.1.x.x network )<br>
380  *
381  * Explanation: The RF24Gateway/RF24Mesh network supports about 253 hosts. I setup the RF24Gateway and all associated
382  * devices with a much larger subnet in this config to encompass all devices directly accessible via the RF24Gateway network.
383  *
384  * ie: RPi master 10.1.3.1/16, RPi2 child 10.1.3.33/16, RPi2 child usb0 10.1.5.1/24
385  *
386  * Routing Configuration:
387  *
388  * In the RF24Gateway example working directory, create a file "routing.txt" <br>
389  * Add routing entries in a standard kind of format: <br>
390  * ie: IP<space>NetMask<space>Gateway <br>
391  * Example routing.txt file:
392  *
393  * 10.1.5.0 255.255.255.0 10.1.3.33 <br>
394  * 10.1.4.0 255.255.255.0 10.1.3.34 <br>
395  * 0.0.0.0 0.0.0.0 10.10.3.33 <br>
396  *
397  * First entry: Traffic for 10.1.5.x will use 10.1.3.33 as the gateway <br>
398  * 2nd entry: Traffic for 10.1.4.x will use 10.1.3.34 as the gateway <br>
399  * 3rd entry: Traffic not in the RF24Gateway network will be sent via 10.10.3.33 by default
400  *
401  * Additional Routing Configuration:<br>
402  * You can specify any combination of IP/Netmask in the routing.txt file to create unique subnets for routing traffic:
403  *
404  * 10.1.5.0 255.255.255.224 10.1.3.33 <br>
405  * 10.1.5.32 255.255.255.224 10.1.3.34 <br>
406  * 10.1.5.64 255.255.255.224 10.1.3.35 <br>
407  * 0.0.0.0 0.0.0.0 10.10.3.32 <br>
408  *
409  * First entry: Traffic for 10.1.5.(1-30) will use 10.1.3.33 as the gateway <br>
410  * 2nd entry: Traffic for 10.1.5.(33-62) will use 10.1.3.34 as the gateway <br>
411  * 3rd entry: Traffic for 10.1.5.(65-94) will use 10.1.3.35 as the gateway <br>
412  * 3rd entry: Traffic not in the RF24Gateway network will be sent via 10.10.3.32 by default <br>
413  *
414  *
